# Vigil Project Constitution (AGENTS.md)

This document serves as the project constitution and styling guidelines for the Vigil platform. All agents operating within this workspace must adhere strictly to these rules.

---

## 1. Tech Stack & Libraries

### Backend (Python)
- **Agent Orchestration**: `langgraph` (v0.1+)
- **LLM Gateway**: `portkey-ai` (v0.1+) using Groq as the primary engine. The standard primary reasoning model slug is **`groq/llama-3.3-70b-versatile`** (with configured fallbacks).
- **Vision/OCR Integration**: `openrouter` API for free-tier vision models (primarily `nvidia/nemotron-nano-12b-v2-vl:free` for scanned assets)
- **Local Document Parsers**: `pymupdf` (primary/faster parser for native PDFs), `pdfplumber` (fallback parser for native PDFs), `unstructured`, and `python-docx` (for text-native documents)
- **Vector Storage & Embeddings**: `qdrant-client` + `fastembed` (standard embedding model: **`BAAI/bge-small-en-v1.5`**)
- **Reranking**: `flashrank`
- **Evaluation**: `ragas`
- **Observability**: `langsmith` (configured and actively tracing)

### Knowledge Storage (OKF Bundle)
- **Format**: Open Knowledge Format (OKF v0.1)
- **Structure**: A local file directory containing Markdown files (`.md`) with YAML frontmatter, cross-linked via relative markdown links.
- **Index and Log files**: Every directory requires an `index.md` (to map sub-concepts/files) and a `log.md` (audit trail of ingestion, updates, and links).

### Frontend (Next.js)
- **Framework**: Next.js (App Router)
- **Theme & Styling**: Tailwind CSS with `next-themes` (for light/dark toggling)
- **Animations**: `framer-motion`
- **Data Visualization**: `react-force-graph-2d` (for live 2D knowledge graph interactions, Obsidian-style force layout)

## 2. OKF Frontmatter Schema & Linking

Every concept file must be a Markdown file with a YAML frontmatter.

### Schema
```yaml
---
type: concept | procedure | regulation | maintenance_log | drawing | alert
title: "Descriptive Title of the Entity/Concept"
description: "A detailed but concise summary of the concept and its significance."
resource: "Relative path to original source document, external URL, or pipeline session ID (for alert type)"
tags: [tag1, tag2, tag3]
timestamp: 2026-07-04T09:15:38+05:30
# Optional fields (required for type: alert)
confidence_score: 0.94 # Float from 0.0 to 1.0
severity: low | medium | high | critical
---
```

### Linking Conventions
- Cross-link related concepts using relative markdown links: `[Target Concept](./target_concept.md)`.
- Use explicit labels instead of generic link texts.
- Do not use absolute paths or hardcoded local file paths for links.

## 3. System Architecture & Agent Responsibilities

Vigil's architecture consists of one central Ingestion Pipeline and four distinct LangGraph-based Query Agents.

### A. Central Ingestion Pipeline
1. **Ingestion Pipeline**: Orchestrates document parsing (local/OCR), entity extraction, OKF concept creation, contradiction check, and semantic embedding.
   - **Contradiction Detection (Pipeline Step)**: Post-ingestion verification step that evaluates the newly written concept against linked OKF concepts for conflicts.

### B. The 4 Query Agents (LangGraph-based)
1. **Expert Copilot Agent**: Provides interactive GraphRAG-based chat with users, answering technical queries and providing precise document citations.
2. **Maintenance & RCA Agent**: Analyzes maintenance history, logs, and sensor alerts to run Root Cause Analysis (RCA) and recommend corrective procedures.
3. **Compliance Agent**: Compares operational procedures against regulatory documents to identify violations, missing steps, and compliance risks.
4. **Lessons-Learned Engine**: Extracts key insights, failures, and optimization patterns from maintenance history and updates the knowledge graph with synthesized lessons.

## 4. Hard Rules

1. **No External NIM Downloadable Containers**: Never invoke NIM's downloadable container paths. All vision and OCR tasks must route through **OpenRouter** APIs.
2. **No Hardcoded Secrets/Keys**: API keys (Groq, OpenRouter, Portkey, Qdrant, LangSmith, etc.) must NEVER be hardcoded. Retrieve them from `.env` environment variables using `os.getenv()`.
3. **Valid OKF Enforcement**: Every document ingested must produce a valid OKF Markdown file conforming to the schema, and register itself in the corresponding directory's `index.md` and `log.md`.
4. **Traceability & Citations**: Every output generated by an agent must be directly traceable to a source OKF concept or document. Unverifiable statements (hallucinations) are strictly prohibited.
5. **Timer & Task Management**: Never use `sleep` commands in code to wait for jobs. Use scheduling utilities, cron configurations, or async hooks.
6. **Placeholder API Key Fallbacks**: If the primary Portkey or Groq API keys are set to placeholder values, the system must fall back to routing all LLM calls directly through the OpenRouter endpoint using the configured free or paid models.

### Contradiction-Detection Mechanism
- Upon writing a new OKF concept, the ingestion pipeline must:
  1. Retrieve all directly linked concepts referenced via markdown links in its frontmatter and body.
  2. Perform an LLM-based evaluation querying if the newly ingested concept's content directly contradicts or conflicts with the content of each linked concept.
  3. If a contradiction is detected (with a model confidence score > 0.7), the system must generate a new alert file in the dedicated `alerts/` directory as its own OKF concept (e.g. `alerts/conflict_xyz.md`). This alert file must contain the description of the conflict and explicitly link back to both source conflicting concepts.

### OpenRouter API Fallbacks
- During scanned document parsing/OCR, if the primary free-tier vision model (`nvidia/nemotron-nano-12b-v2-vl:free`) is rate-limited or unavailable:
  1. Catch the exception/HTTP 429 status code.
  2. Fall back to secondary free-tier models via OpenRouter (specifically `google/gemma-4-26b-a4b-it:free` or `openrouter/free`).
  3. Only raise a pipeline error if all fallback models fail or return errors.

### RAGAS Evaluation Compatibility
- All agent interfaces, user interactions, and query responses must log structured data compatible with the RAGAS framework.
- The pipeline must capture and format every retrieval interaction as a structured tuple:
  - `question`: The user query or evaluation prompt.
  - `contexts`: An array of text snippets/chunks retrieved from OKF files and Qdrant.
  - `answer`: The generated response from the agent.
- Log these triplets to a structured JSONL format in a dedicated evaluation log directory for automated suite consumption.

## 5. Code Style & Conventions

### General Text & Formatting
- **No Em Dashes**: Never use em dashes (—) anywhere in this project - not in code comments, docstrings, READMEs, commit messages, or any generated markdown documentation. Use regular hyphens, colons, or restructure the sentence instead. An exception is allowed for raw, verbatim external logs or LLM-generated outputs in logs/ or interactions.jsonl, which should not be modified.

### Python (Backend)
- **Linting & Formatting**: Follow PEP 8 guidelines. Use `black` and `isort`.
- **File Limits**: Keep individual Python files under **400 lines**. Separate logic into modules (e.g., `nodes/`, `state/`, `utils/`).
- **Typing**: Use strict type hinting (`from typing import ...`) for all function signatures and LangGraph states.

### TypeScript / Next.js (Frontend)
- **Formatting**: Use Prettier and ESLint.
- **Component File Limits**: Keep UI component files under **250 lines**. Break complex sections into sub-components.
- **State Management**: Keep UI state local or use React Context. Avoid bloated global store patterns unless absolutely necessary.

## 6. Git Commit Conventions

All commits must follow the **Conventional Commits** specification:

- `feat`: A new feature (e.g., `feat(ingest): add unstructured pdf parser`)
- `fix`: A bug fix (e.g., `fix(copilot): resolve missing citation bug`)
- `docs`: Documentation changes (e.g., `docs(readme): update project setup`)
- `style`: Changes that do not affect the meaning of the code (white-space, formatting, missing semi-colons, etc.)
- `refactor`: A code change that neither fixes a bug nor adds a feature
- `perf`: A code change that improves performance
- `test`: Adding missing tests or correcting existing tests
- `chore`: Updates to build processes, auxiliary tools, or library dependencies
